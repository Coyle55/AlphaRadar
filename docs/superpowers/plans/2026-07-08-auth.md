# Auth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Signup/login/logout as an API surface (no web UI — that's Plan 3c), backed by the local Supabase Auth service already running since Plan 1, with an app-specific `profiles` row auto-created for every user.

**Architecture:** Supabase Auth owns credential storage in its own managed `auth.users` table. Next.js integrates via `@supabase/ssr`, which manages the session as an httpOnly cookie through `next/headers`. A Postgres trigger on `auth.users` auto-creates a matching `public.profiles` row for every new user, so the two tables can never drift out of sync. Three route handlers (`signup`, `login`, `logout`) and a `getCurrentUser` server helper (exposed via a thin `GET /api/auth/me` route) are the full surface this plan delivers — Plan 3b's position routes will import `getCurrentUser` directly.

**Tech Stack:** Same as Plan 1/2 — Next.js 16 (App Router) + TypeScript, local Supabase (Postgres + Auth), Vitest. New: `@supabase/ssr`, `@supabase/supabase-js`.

## Global Constraints

- Supabase Auth owns credential storage and verification; this codebase never hashes or stores passwords itself.
- Login failures return a generic `"invalid email or password"` — never distinguish "no such user" from "wrong password" in the response, to avoid leaking which emails are registered.
- Every `auth.users` row must have a corresponding `profiles` row. This is enforced by a Postgres trigger (`on_auth_user_created`), not application code — there is no code path that creates a user without going through Supabase Auth's signup, which the trigger always fires from.
- Route handler tests mock `@/lib/supabase/server`'s `createClient` entirely rather than exercising real cookie-based sessions. `next/headers`'s `cookies()` requires genuine Next.js request-scoped context that direct unit-test invocation of a route handler doesn't provide — this is a real constraint, not a testing shortcut. Real session behavior (cookies actually persisting across requests, the trigger actually firing) is verified via live manual testing (`curl` with a cookie jar), the same pattern already used for Telegram delivery in Plan 2.
- Node 23.x, TypeScript strict mode, Next.js App Router only.

- [ ] **Step 1: Write the migration**

```sql
-- supabase/migrations/0003_profiles_table.sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  telegram_chat_id text,
  subscription_tier text not null default 'free',
  created_at timestamptz not null default now()
);

create function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  insert into public.profiles (id) values (new.id);
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

// src/lib/supabase/server.ts
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export async function createClient() {
  const cookieStore = await cookies();

  return createServerClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!, {
    cookies: {
      getAll() {
        return cookieStore.getAll();
      },
      setAll(cookiesToSet) {
        try {
          cookiesToSet.forEach(({ name, value, options }) => {
            cookieStore.set(name, value, options);
          });
        } catch {
          // setAll was called from a context that can't set cookies (e.g. a
          // Server Component during render). No middleware exists yet in
          // this plan — every caller in Tasks 4-7 is a route handler, where
          // setAll always succeeds — so this catch is a defensive no-op,
          // not something expected to trigger in this plan's code paths.
        }
      },
    },
  });
}
```

// src/app/api/auth/signup/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { createClient } from '@/lib/supabase/server';

export async function POST(request: NextRequest) {
  const { email, password } = await request.json();

  if (!email || !password) {
    return NextResponse.json({ error: 'email and password are required' }, { status: 400 });
  }

  const supabase = await createClient();
  const { data, error } = await supabase.auth.signUp({ email, password });

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 400 });
  }

  return NextResponse.json({ userId: data.user?.id ?? null }, { status: 201 });
}
```

import { createClient } from '@/lib/supabase/server';

export interface CurrentUser {
  id: string;
  email: string | null;
}

export async function getCurrentUser(): Promise<CurrentUser | null> {
  const supabase = await createClient();
  const { data, error } = await supabase.auth.getUser();

  if (error || !data.user) return null;

  return { id: data.user.id, email: data.user.email ?? null };
}
